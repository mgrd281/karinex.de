## 1. Quick start (first 30 seconds)

**What you're working on:** a custom Shopify Online Store 2.0 theme for
karinex.de — a German online shop selling Microsoft / Windows /
Visual Studio license keys.

**Before you write a line of code:**

1. Branch off `origin/main`. Never commit directly to `main`.
2. The merchant ships to production by merging to `main`. Shopify
   auto-syncs within ~1–2 minutes — there is no staging.
3. No build, no bundler, no jQuery, no npm install. You edit Liquid
   / CSS / JS files directly and they appear on the live site.
4. **Brand colours are sacred:** deep green `#1D4739` (primary),
   champagne gold `#c9a84c` (accent only — never primary).
5. **Test in Incognito** after merge. Shopify CDN caches aggressively.

## 2. Critical don'ts

| Don't | Why | What to do instead |
|---|---|---|
| Don't edit `config/settings_data.json` | Shopify Theme Editor overwrites it | Ask the merchant to change it in Admin |
| Don't edit `config/settings_schema.json` | Same | Same |
| Don't edit `templates/*.json` casually | Auto-generated by Theme Editor | Only when adding section types the editor can't reach; preserve the `/* This file is auto-generated… */` header |
| Don't use `{{ routes.root_url }}` in `href` | On `/en/`, `/pl/`, `/sv/` it becomes `/enpages/…` (404) | Use the `_root` pattern — see [§7 Liquid](#liquid) |
| Don't change Crisp to `display: none` | Breaks `message:received` callbacks | Hide off-screen (`left: -10000px`) — see [§17](#17-key-integrations--gotchas) |
| Don't `npm install` anything | There is no build step | Ship readable, ES5-friendly JS |
| Don't push to `main` directly | All work goes through PRs | Branch + PR + squash-merge |
| Don't force-push, `git reset --hard`, or `git clean -fd` | History is shared | Use `git revert` instead |
| Don't translate strings inline with `{% if locale == 'en' %}` per phrase | Hard to maintain | Use top-of-file `{% case %}` or `{{ 'key' \| t }}` against `locales/*.json` |
| Don't add jQuery, React, or any framework | Theme is vanilla on purpose (perf, cost) | Web Components or plain DOM |

## 3. What this repo is

A **custom Shopify Online Store 2.0 theme** for the karinex.de
shop. The theme is heavily customised on top of a Shopify base
theme — most of the visible UI in the brand is in `sections/`,
`snippets/`, and `assets/`.

The shop sells **digital products only** (license keys, delivered
by e-mail after payment). There is no physical inventory.

The primary visual identity:
- Deep green `#1D4739` for primary surfaces and CTAs
- Champagne gold `#c9a84c` for subtle accents (lines, pinstripes)
- Cream `#faf7f0`, ink `#0d2e22`, off-white `#ffffff`
- Newsreader serif for display + body, Red Hat Display for accent
  uppercase

## 4. Tech stack

| Layer | Choice | Notes |
|---|---|---|
| Templating | Shopify **Liquid** (server-rendered) | Single source of truth. No client-rendered shell. |
| Section/Block model | Shopify Online Store 2.0 | JSON templates compose `.liquid` sections + blocks |
| Styling | Plain CSS | No SCSS, no Tailwind, no PostCSS. Inline `<style>` blocks in sections are normal |
| Scripting | Vanilla **ES5-flavoured JS**, occasional Web Components | **No** jQuery, no bundler, no transpiler. Use IIFEs |
| i18n | Native Shopify `{% form 'localization' %}` + `locales/*.json` | Active locales: **DE (primary)**, EN, PL, SV |
| Live chat | [Crisp](https://crisp.chat) | Website ID `3dc17a60-f1b3-4b84-9cb5-2998ab31396a`. Default UI is hidden off-screen; we wrap it in our gold widget at `snippets/karinex-live-chat.liquid` |
| Consent | Custom cookie banner + Usercentrics CMP | `snippets/karinex-cookie-banner.liquid` and `snippets/smart-script-manager.liquid` |
| Analytics / tracking | Google Tag Manager, Ahrefs Analytics | Loaded lazily by smart-script-manager |
| Trust seals | TrustedShops, Trustami | Footer + legal pages |
| Invoicing | Custom backend at `invoice-production-8cd6.up.railway.app` | Triggered after order completion |
| Deploy | Shopify ↔ GitHub auto-sync | Merging to `main` propagates within ~1–2 minutes. No CI. |
| Local dev | None | Edits go via PR + merge. There is no `localhost`. |

## 5. Glossary

### Technical terms

- **Section** — Shopify-specific unit of UI. A single `.liquid` file
  in `sections/` with its own `{% schema %}`. Configurable by the
  merchant in the Theme Editor.
- **Block** — A reusable, drag-and-drop UI unit inside a section.
  Lives in `blocks/`.
- **Snippet** — A reusable Liquid partial included via
  `{% render 'snippet-name' %}`. Lives in `snippets/`. No `{% schema %}`.
- **JSON template** — A template file (`templates/*.json`) that
  composes sections by id. The Theme Editor manages this.
- **Liquid template** — A template file (`templates/*.liquid`)
  written entirely by us. We use these only when JSON can't.
- **`_root`** — A locale-aware URL prefix we compute at the top of
  every section/snippet that links to internal pages. Always ends
  with `/` (`'/'` on DE, `'/en/'` on EN, etc.). Replaces the raw
  `{{ routes.root_url }}` to avoid the `/enpages/…` 404 bug.
- **`request.locale.iso_code`** — Shopify's two-letter locale of the
  current request (`de`, `en`, `pl`, `sv`). The single source of
  truth for which language UI strings should render in.
- **`localization.country.iso_code`** — Shopify Markets' IP-derived
  country code. Used by the geo-redirect in `layout/theme.liquid`.
- **Crisp Website ID** — `3dc17a60-f1b3-4b84-9cb5-2998ab31396a`.
  The single tenant for the live chat backend.
- **Section ID prefix in JSON templates** — Shopify renders
  `<div id="shopify-section-template--<hash>__<key>">`. Use
  `[id$="__main"]` attribute selectors when targeting from CSS,
  not `#shopify-section-main` (won't match).

### Business / brand terms

- **Karinex Concierge** — The brand's customer-service voice. The
  green gold-accented chat widget UI is named after it.
- **AboutYou template** — An alternative product page template
  inspired by aboutyou.de. Lives in `sections/product-aboutyou.liquid`
  and `templates/product.aboutyou.json`. Used for select products
  the merchant assigns it to.
- **AY utility bar** — The top bar with "Kontakt & Hilfe" +
  locale switcher, AboutYou-styled. `sections/header.liquid`.
- **Premium Service** — Marketing positioning. Appears as a tagline
  in the chat widget footer.
- **Mega menu** — Desktop main-nav dropdown with Office / Windows /
  Server / Tools / Sale categories. `snippets/mega-menu-software-unified.liquid`.

## 7. Code conventions

### Naming prefixes (CSS classes / JS IDs)

Custom-namespaced everywhere to avoid collisions with the Shopify
base theme + third-party scripts.

| Prefix | Owner | Used for |
|---|---|---|
| `klc-` | **K**arinex **L**ive **C**hat | floating chat widget |
| `klsw-` | Karinex Locale SWitcher | locale modal (reserved) |
| `klb-` | Karinex Language Banner | first-visit lang suggestion |
| `kx-` | Karinex (legacy) | older Karinex snippets, cookie banner, loyalty |
| `kr-` | Karinex Rewards | rewards CSS tokens |
| `kx-ly-` | Karinex LoYalty | loyalty page CSS |
| `ay-` | About-You-style | AboutYou-pattern components (utility bar, product) |
| `gc-` | Gift Cards | gift-card page styling |
| `ps-` | Product Software | software-product specific styling |

**Rule:** when you add a new component, always reach for an
existing prefix that fits its purpose. Don't introduce a new prefix
unless it represents a new feature family. If you do, document it
here.

### CSS

- Plain CSS, mobile-first where possible
- CSS custom properties (`--klc-green`, `--rating-star-color`, …)
  scoped to the component, defined at the top of the snippet
- Inline `<style>` blocks inside section/snippet files are normal —
  keeps each snippet self-contained
- Respect `@media (prefers-reduced-motion: reduce)` whenever you add
  animation
- **Brand tokens:**
  - Primary: deep green `#1D4739`
  - Highlights: `#2d6b54`, `#0a2118` (deepest)
  - Accent: champagne gold `#c9a84c` (never primary)
  - Surfaces: cream `#faf7f0`, off-white `#ffffff`
  - Ink: `#0d2e22`
- **Spacing:** prefer multiples of 4 px. Avoid hardcoded magic numbers
  except in single-use micro-positioning.

### JavaScript

- **No build step** — ship readable, ES5-friendly code (no optional
  chaining `?.`, no `??`, no top-level `await`)
- **No jQuery**, no React, no bundler
- Wrap in IIFE:
  ```js
  (function () {
    // ...
  })();
  ```
- DOM lookups: `getElementById` first, fall back to `querySelector`
- Lazy-load third-party scripts on `requestIdleCallback`:
  ```js
  if ('requestIdleCallback' in window) {
    requestIdleCallback(loadScript, { timeout: 2000 });
  } else {
    setTimeout(loadScript, 2000);
  }
  ```
- Bail out early on conditions that should skip the whole IIFE
  (e.g., mobile-only widgets), rather than wrapping the whole body
  in an `if`.

### Liquid

**Always** use `_root` for locale-aware root URL inside any custom
section/snippet that links to internal pages:

```liquid
{%- liquid
  assign _root = routes.root_url | default: '/'
  unless _root == '/'
    assign _root = _root | append: '/'
  endunless
-%}
<a href="{{ _root }}pages/kontakt">Kontakt</a>
```

Why: `routes.root_url` is `'/'` on the default locale but `'/en'`
(no trailing slash) on others. Naïve concatenation produces
`/enpages/kontakt` → 404. This bug has been the single largest
source of broken links on the site.

**Multi-language strings inside custom sections:** use a single
`case` block at the top of the file that assigns translated
variables — don't spread `{% if %}` per phrase.

```liquid
{%- case request.locale.iso_code -%}
  {%- when 'en' -%}
    {%- assign T_hello = 'Hello' -%}
  {%- when 'pl' -%}
    {%- assign T_hello = 'Cześć' -%}
  {%- else -%}
    {%- assign T_hello = 'Hallo' -%}
{%- endcase -%}

<h1>{{ T_hello }}</h1>
```

For theme-editor-translatable strings (shown in `locales/*.json`):

```liquid
<h1>{{ 'general.hello' | t }}</h1>
```

### Branch / PR strategy

- **Branch off `origin/main`** with a kebab-case name describing the
  intent: `fix-…`, `feat-…`, `chore-…`, `restyle-…`, `docs-…`
- **One topic per PR** — keeps merges atomic and easy to revert
- **Conventional commit message:** `feat(<scope>): …`, `fix(<scope>): …`
- **Squash-merge** to `main`; Shopify auto-syncs from `main`
- **Reverting a regression:**
  ```bash
  git fetch origin main
  git checkout origin/main -b revert-<sha>
  git revert --no-edit <merge-sha>
  git push -u origin revert-<sha>
  # then open a PR + squash-merge
  ```

## 8. Common workflows (copy-pasta)

### 8.1 Add a new section

1. Create `sections/<name>.liquid` with:
   ```liquid
   {%- comment -%} Brief description of what this section does {%- endcomment -%}

   <div class="kx-<name>">
     <!-- markup -->
   </div>

   <style>
     .kx-<name> { /* … */ }
   </style>

   {% schema %}
   {
     "name": "Section title",
     "settings": [],
     "presets": [{ "name": "Section title" }]
   }
   {% endschema %}
   ```
2. (Optional) Move CSS to `assets/<name>.css` and reference via
   `{{ '<name>.css' | asset_url | stylesheet_tag }}`
3. Commit + PR + merge → merchant can drag it into any page from
   the Theme Editor

### 8.2 Add a translated string

For UI strings managed by Theme Editor / locales:

1. Edit `locales/en.default.json` and add:
   ```json
   "general": { "hello": "Hello" }
   ```
2. Add matching keys to `locales/de.json`, `locales/pl.json`,
   `locales/sv.json` (and any others that exist)
3. Reference in Liquid: `{{ 'general.hello' | t }}`

For UI strings only used inside one section: use the inline `case`
pattern in [§7 Liquid](#liquid).

### 8.3 Add a new locale (e.g., French)

1. Create `locales/fr.json` (copy keys from `de.json` and translate)
2. Create `locales/fr.schema.json` (copy from `de.schema.json`)
3. Add to the geo-redirect table in `layout/theme.liquid`:
   ```liquid
   {%- elsif _geo_country == 'FR' or _geo_country == 'BE' or _geo_country == 'LU' -%}
     {%- assign _expected = 'fr' -%}
   ```
4. The Shopify Admin → Settings → Languages must also enable French
   for visitors to see it at runtime — code alone is not enough.

### 8.4 Revert a broken merged PR

```bash
git fetch origin main
git checkout origin/main -b revert-<short-name>
git revert --no-edit <merge-sha>
git push -u origin revert-<short-name>
# Open PR via gh / GitHub UI / mcp tool, then squash-merge
```

If multiple PRs need reverting, revert in **reverse order** (newest
first) to keep the diffs clean.

### 8.5 Verify a change end-to-end

1. Wait ~1–2 minutes after squash-merge for Shopify to pull `main`
2. Open `https://www.karinex.de/` in an **incognito** window
3. Hard-refresh (Cmd+Shift+R / Ctrl+F5)
4. For mobile: real device, or Chrome / Safari DevTools device emulator
5. For SEO changes: re-run an Ahrefs site audit ~24 h later

### 8.6 Debug a layout bug

1. Reproduce in incognito
2. Open DevTools → Elements → inspect the broken element
3. Look at the matched section by `id="shopify-section-…"` — that
   tells you which `.liquid` file rendered it
4. Toggle styles in the Computed pane until you find the offending
   rule, then trace it back to a `<style>` block in the section

### 8.7 Trigger Shopify sync manually if it stalled

1. Shopify Admin → Online Store → Themes
2. Click "..." on the live theme → **Edit code**
3. Save any file (no actual change needed) — Shopify re-pulls
4. Wait 30 s and verify

### 8.8 Add a Shopify URL Redirect (bulk)

1. Edit `docs/shopify-redirects.csv` (format: `from,to`, one per line)
2. Shopify Admin → Online Store → Navigation → URL Redirects → **Import**
3. Upload the CSV → "Upload and continue" → Import

### 8.9 Translate a hardcoded German string

Find the offending file (usually a Karinex-custom section/snippet):

```bash
grep -rn 'Verfügbarkeit\|Artikelnummer\|Blitz-Versand' sections/ snippets/
```

Replace with `{{ 'key' | t }}` and add the key to all 4 locale JSON
files (DE / EN / PL / SV).

## 9. Anti-patterns from real PR history

These mistakes have actually happened on this repo. Avoid them.

### 9.1 The `/enpages/` 404 flood (PR #95, #97)

```liquid
❌  <a href="{{ routes.root_url }}pages/impressum">Impressum</a>
✅  <a href="{{ _root }}pages/impressum">Impressum</a>
```

On `/en/` locales `routes.root_url == '/en'` (no trailing slash) and
concatenation produces `/enpages/impressum` → 404. The cookie banner
emitted this on **every page** for months, generating 295+ broken
inbound links in Ahrefs.

### 9.2 Locale-stacking in hreflang (PR #96)

```liquid
❌  if _path contains '/en/'   ...strip...
    if _path contains '/pl/'   ...strip...
    # /sv/, /da/, /nl/ etc. fall through → emit /en/sv/page (404)

✅  # Strip ANY known locale prefix from request.path before
    # building hreflang URLs:
    assign _path_seg = _hreflang_path | split: '/'
    assign _path_locale = _path_seg[1] | default: ''
    assign _known_locales = 'en,pl,sv,da,nl,fr,it,es,cs,hu,ro,sk,fi,el,pt' | split: ','
    if _known_locales contains _path_locale
      assign _strip = '/' | append: _path_locale
      assign _hreflang_path = _hreflang_path | remove_first: _strip
    endif
```

### 9.3 `display: none` on Crisp (PR #100)

```css
❌  #crisp-chatbox { display: none !important; }
✅  #crisp-chatbox {
      position: fixed !important;
      left: -10000px !important;
      top: -10000px !important;
      width: 1px !important;
      height: 1px !important;
      opacity: 0 !important;
      pointer-events: none !important;
    }
```

`display: none` removes the chatbox from layout. Crisp's WebSocket
listener queue stops dispatching `message:received` → operator
replies never appear in our UI.

### 9.4 Wrong selector for JSON-template section IDs (PR #119)

```css
❌  #shopify-section-main_mobile { display: none; }
✅  [id$="__main_mobile"][class*="shopify-section"] { display: none; }
```

Shopify wraps JSON-template sections with IDs like
`shopify-section-template--<hash>__main_mobile`. The bare ID never
matches.

### 9.5 Forgetting to clear localStorage flags during testing

The smart language banner (PR #106) sets `karinex_lang_pref` after
first interaction. If you're testing it repeatedly, run:

```js
localStorage.removeItem('karinex_lang_pref');
```

in DevTools console before each reload.

### 9.6 Two `<style>` blocks fighting over the same `!important`

Order matters when both blocks use `!important`. Inline `<style>`
inside a section loses to a later `<style>` in the same document.
Solution: put the override in `layout/theme.liquid` (loaded first)
or increase specificity instead of stacking importance.

### 9.7 Pushing translations to JS without exposing them via Liquid

```js
❌  alert('Bitte wählen Sie ein Bild aus.');  // hardcoded
✅  alert(KLC_I18N.errImageVideo);             // populated via Liquid
```

Add the translated string to the Liquid `case` block at the top
of the file and surface it on `window.<NAMESPACE>_I18N = { … }`.

### 9.8 Cargo-culting `routes.root_url` from Shopify Dawn

Shopify Dawn ships with `{{ routes.root_url }}{{ product.url }}`
patterns that work because `product.url` already starts with `/`.
Our links rarely use `product.url` — they use string suffixes
like `pages/x`, so we need `_root` with trailing slash.

### 9.9 Editing `templates/product.json` and breaking Theme Editor

You CAN add new sections to JSON templates (PR #118 did this), but:
- Preserve the `/* auto-generated */` comment header
- Don't reorder the existing sections array
- Don't rename existing keys
- The merchant should still be able to drag-arrange sections in
  the Theme Editor afterwards

## 10. Critical paths

### 10.1 First visit (cold visitor)

```
GET /
  layout/theme.liquid
    └─ geo-redirect (line 11-50): IP → locale → maybe 302 to /en or /pl or /sv
    └─ smart-script-manager: queue UC + GTM
    └─ render snippets:
       - karinex-lang-suggestion-banner (sticky top, if locale mismatch)
       - karinex-cookie-banner (bottom, if no consent cookie)
       - karinex-live-chat (bottom-right gold bubble)
    └─ render section header.liquid
       - utility bar
       - mega menu / mobile menu
       - cart icon
    └─ template body (sections from JSON template)
    └─ custom-footer.liquid
```

### 10.2 Locale switch (manual via dropdown)

```
User opens header → locale dropdown
  → renders snippets/localization-form.liquid
  → user picks new country → submits {% form 'localization' %}
  → Shopify sets locale cookie + reloads page on chosen locale
  → geo-redirect at top of theme.liquid sees matching cookie + skips
```

### 10.3 Add-to-cart → checkout

```
sections/product-information.liquid (or product-aboutyou.liquid)
  → user picks variant → clicks "In den Warenkorb"
  → form POST /cart/add → Shopify creates line item
  → cart drawer opens (snippets/cart-drawer.liquid)
  → user clicks "Checkout" → Shopify-hosted checkout flow (we don't own this)
  → on order completion → Shopify webhook → custom invoice backend
```

### 10.4 Live chat first message

```
User clicks gold bubble → snippets/karinex-live-chat.liquid panel opens
  → user fills prechat wizard (Sind Sie Kunde? + Thema + Name + Email + GDPR)
  → submit → loadCrisp() injects client.crisp.chat/l.js
  → $crisp.push(['set', 'user:email', …])
  → $crisp.push(['do', 'message:send', ['text', context]])
  → $crisp.push(['on', 'message:received', cb])  ← handler renders agent bubbles
  → agent reply arrives via WebSocket → cb fires → cream bubble in panel
```

### 10.5 Page-has-broken-links detection (SEO cycle)

```
Merchant runs Ahrefs site audit (~daily)
  → Ahrefs crawls karinex.de + emits CSV of 404 URLs
  → merchant uploads CSV to this session
  → claude inspects + groups by root cause (locale bug, dead handle, etc.)
  → fix via PR for code-side causes
  → docs/shopify-redirects.csv for legacy URLs → merchant imports in Admin
  → next Ahrefs cycle confirms reduction
```

## 11. Architecture decision records

Brief, dated, irreversible-ish decisions and why we made them.

### ADR-001: No build step (May 2026)

**Decision:** Theme ships raw Liquid / CSS / JS. No webpack, no
Rollup, no transpiler.

**Why:** Shopify's GitHub integration pulls files as-is into the
live theme. Adding a build pipeline would require GitHub Actions
that build into a deploy branch — workable but complicates rollback
and creates a source-vs-deployed delta the merchant can't read in
the Shopify Code Editor.

**Cost:** No TypeScript, no modern JS features that don't run in
ES5, no CSS variables polyfilling for legacy IE.

### ADR-002: Custom cookie banner instead of Shopify's default (May 2026)

**Decision:** We render `snippets/karinex-cookie-banner.liquid`
and hide `shopify-privacy-banner` via CSS.

**Why:** Shopify's banner is functional but visually generic.
Karinex needed a luxury, brand-consistent banner with custom CTA
copy and German legal text.

**Cost:** We own the consent UX surface and must keep it GDPR-compliant
ourselves.

### ADR-003: Crisp as the chat backend, custom widget on top (May 2026)

**Decision:** Use Crisp for transport; build our own gold KARINEX
widget as the visible UI. Crisp's default chatbox is moved off-screen.

**Why:** Other competitors (Tidio, Intercom, Drift) either have
worse free tiers or less polished APIs. Crisp's `$crisp.push` event
bus is well-documented. Their default UI doesn't match Karinex
brand, so we wrap it.

**Cost:** Operator messages are delivered to our `message:received`
callback only when the chat is in `open` state — see [§17](#17-key-integrations--gotchas)
for the workaround.

### ADR-004: Plain CSS in `<style>` blocks per section (May 2026)

**Decision:** Most styles live inline in the section/snippet that
owns them.

**Why:** Keeps each section self-contained (you can move a section
to another theme without missing CSS). Avoids the cascading-rules
nightmare of a single mega-stylesheet.

**Cost:** Slightly more HTML weight per page. Easier debugging.

### ADR-005: AboutYou alt template (May 2026)

**Decision:** Keep two product templates side by side
(`product-information.liquid` = Karinex green concierge,
`product-aboutyou.liquid` = AboutYou-style hero+badges).
Merchant assigns per product.

**Why:** Different product categories benefit from different
layouts. Office bundles use AboutYou-style imagery; software-only
SKUs use the green concierge.

### ADR-006: SEO 404 cleanup via combination of code fixes + bulk redirects (May 2026)

**Decision:** Fix root-cause bugs in code (locale concat, hreflang
stacking) AND publish a `docs/shopify-redirects.csv` for the merchant
to bulk-import.

**Why:** Code fixes prevent new 404s; redirects catch historical
inbound links that crawlers still hit.

### ADR-007: Swedish (`/sv/`) is a first-class indexable locale (June 2026)

**Decision:** `/sv/` pages render `index, follow` and participate in the
hreflang cluster (`hreflang="sv"`), exactly like `/en/` and `/pl/`.

**History:** For a brief window (May 2026) every `/sv/` product /
collection / article / blog was blanket-`noindex`ed and excluded from
hreflang, on the assumption that Karinex did not serve a Swedish market.
That created the opposite Ahrefs warning — **"Noindex page in sitemap"**
(108 URLs) — because Shopify still lists `/sv/` in its auto-generated
Swedish sitemap (Swedish is a *published language*; the theme cannot edit
that sitemap). The merchant confirmed they **do** want Swedish traffic and
the Swedish storefront, so the noindex was removed instead of unpublishing
the language.

**Why this is safe:** every `/sv/` URL returns HTTP 200, and Swedish reuses
the **German product handles** (no per-locale translated handles), so the
path-based `/sv/<de-handle>` URLs are 200 OK — no hreflang-to-redirect
chains (the May regression does not recur).

**Cost / caveat:** much of the `/sv/` *content* is still German. The pages
are now SEO-*eligible*, but won't rank well in Swedish until the product /
collection text is translated (Shopify Admin → **Translate & Adapt**).
Indexing German text under `/sv/` risks Google ignoring the `sv` hreflang.

**Don't:** re-add a blanket `/sv/` `noindex` (in `snippets/meta-tags.liquid`)
or `/sv/` `Disallow` rules (in `templates/robots.txt.liquid`) without
reversing this decision — that brings back the noindex-in-sitemap warning.

### ADR-008: Hreflang is emitted natively by Shopify, not hand-built (June 2026)

**Decision:** Rely on Shopify's native `hreflang` tags from
`content_for_header`. Removed the hand-built hreflang block in
`snippets/meta-tags.liquid` **and** the suppression in
`layout/theme.liquid` that had been rewriting Shopify's
`rel="alternate" hreflang=` to `rel="not-alternate"`.

**Why:** product handles are translated per locale (Admin → Translate &
Adapt), and the translations are **mixed per-product AND per-locale** —
e.g. `/sv/` uses the German handle for `visual-studio-2026` but the
translated handle `kop-…` for `office-2024-professional-plus`. A manual
block (even with `custom.handle_de/en/pl` metafields) cannot know every
locale's real handle, so it emitted URLs that 301-redirect → Ahrefs
**"Hreflang to redirect or broken page"**, and it only listed a hardcoded
subset of locales → **"Missing reciprocal hreflang (no return-tag)."**
Shopify's native hreflang is the only source that knows each product's
real per-locale handle; it lists **every published language** with a
correct self-referencing tag + `x-default`, and is self-maintaining for
new products/languages.

**Caveat:** native hreflang covers *all* published languages, including
the untranslated extras (`da, el, nl, pt-PT`) whose pages still show
German content (duplicate-content risk). To take those out of the cluster,
**unpublish them** in Admin → Settings → Languages (the merchant chose to
keep them for now). The canonical tag is still emitted manually in
`meta-tags.liquid` (Shopify's canonical remains suppressed) — only the
hreflang suppression was removed.

**Don't:** re-add a manual hreflang block in `snippets/meta-tags.liquid`,
and don't re-enable the hreflang suppression in `layout/theme.liquid`.

## 12. Performance budgets

Targets we try to stay inside on a typical product page.

| Metric | Budget | Notes |
|---|---|---|
| HTML weight (transferred) | ≤ 150 KB | Inline CSS counts. Use lazy-loading where possible |
| Critical CSS (inline) | ≤ 20 KB | In `snippets/critical-css.liquid` |
| JS executed before interaction | ≤ 60 KB | Third-party (UC, GTM, Crisp) loaded on `requestIdleCallback` |
| LCP element | ≤ 2.5 s | Hero image: `loading="eager"` + correct `width`/`height` |
| Crisp script load | After idle | Never block initial render |
| Web Components hydration | ≤ 100 ms each | Don't bundle them; ship as plain `<script>` |

When in doubt: defer it.

## 13. Security, privacy & GDPR

The merchant is in Germany — full GDPR applies.

### Consent gates

- **Usercentrics CMP** (loaded by smart-script-manager) is the legal
  basis for storing cookies. Don't ship third-party tracking that
  bypasses it.
- **Cookie banner** is rendered server-side in
  `snippets/karinex-cookie-banner.liquid`. The user's choice is stored
  in `localStorage` (key `karinex_cookie_consent`) and a cookie that
  Usercentrics manages.

### What we collect

- Cart contents, order history, billing data → Shopify managed
- Live chat transcripts → Crisp managed (their server, EU region)
- Newsletter signup → Shopify Email
- IP for geo-redirect → Shopify Markets only; we don't store it
  ourselves
- Browser language for the smart banner → never persisted server-side

### Code rules

- Never log PII to `console.log` in production code
- Never put product keys / license codes in URL parameters
- Never auto-execute customer-supplied HTML
- Strip suspicious tags in user-generated content (e.g., reviews)
  — but we don't accept any UGC yet, so this is preventive
- Outbound requests to non-Shopify origins should be visible in the
  network panel for transparency

### Sensitive endpoints

- `account.karinex.de` — Shopify customer accounts (separate origin)
- `invoice-production-8cd6.up.railway.app` — internal invoicing.
  Webhook only; nothing sends a user there in the browser.

## 15. How to run / deploy

There is **no build step** and no local server. Workflow:

1. **Branch off `origin/main`** (`git checkout origin/main -b <name>`)
2. **Edit** `.liquid`, `.json`, `.css`, `.js` directly
3. **Commit** with a conventional message (`feat:`, `fix:`, `chore:`, `restyle:`)
4. **Push** the branch, **open a PR** against `main`
5. **Squash-merge** when satisfied (no CI configured)
6. Shopify GitHub integration **auto-syncs** `main` → live theme within ~1–2 minutes
7. **Verify in incognito** (Shopify CDN caches aggressively)

### Verifying changes

- **Desktop:** `https://www.karinex.de/` in an incognito window
- **Mobile:** real device, or Chrome / Safari DevTools device emulator
- **DevTools console** is the only debugger
- **SEO changes:** re-run an Ahrefs site audit ~24 h later

### Rolling back

If a merged PR breaks something:

```bash
git fetch origin main
git checkout origin/main -b revert-<short>
git revert --no-edit <merge-sha>
git push -u origin revert-<short>
# Open PR + squash-merge
```

Shopify will pull the revert and re-render the live site within
1–2 minutes.

## 17. Key integrations & gotchas

### Geo-redirect (`layout/theme.liquid:11-50`)

Shopify Markets exposes the visitor's country as
`localization.country.iso_code`. We map that to a locale:

| Country | Locale | URL prefix |
|---|---|---|
| DE / AT / CH / LI / LU | `de` | `/` (no prefix) |
| PL | `pl` | `/pl/` |
| SE | `sv` | `/sv/` |
| anywhere else | `en` | `/en/` |

The redirect is **client-side** (a `<script>` in `<head>`) and is
gated by a `karinex_lang_set` cookie so it doesn't fire on every
page load.

### Crisp chat (`snippets/karinex-live-chat.liquid`)

- Crisp's DOM is hidden off-screen, **not** `display: none` — see
  [§9.3](#93-display-none-on-crisp-pr-100)
- We push `chat:open` after load so `message:received` fires
- We additionally **poll** `chat:messages` every 3.5 s as a
  defensive fallback if events get dropped
- Sets `user:email`, `user:nickname`, and `session:data` from the
  prechat wizard so the operator sees the customer's context
  immediately

### Cookie banner (`snippets/karinex-cookie-banner.liquid`)

- Hides Shopify's `shopify-privacy-banner` via CSS so only our UI shows
- Stores consent in `localStorage` (`karinex_cookie_consent`)
- Renders on every page (don't gate behind template type)

### AboutYou template

- `templates/product.aboutyou.json` + `sections/product-aboutyou.liquid`
- Activated per product via Shopify Admin → Product → **Theme template**
- Section is fully responsive; safe on desktop and mobile

### Auto-generated comment header

Any JSON file starting with `/* This file is auto-generated… */` is
owned by the Theme Editor. When you must edit it (e.g., adding a new
section type), preserve the header byte-for-byte.

### Locale-aware URLs

Never concatenate `routes.root_url` with a path that already starts
with `/`. Use the `_root` pattern in [§7](#liquid). Repeated regressions
of this bug have been the single largest source of 404s.
